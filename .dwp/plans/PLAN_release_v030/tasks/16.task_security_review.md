# Task 16: Security Review

## Context

Review of all v0.3.0 changes for security vulnerabilities.

## Scope

### A. Invite Flow
- Token generation: randomness, brute-force resistance
- Expiry: proper time-based enforcement
- Max uses: counter correctness
- Race condition: token used twice simultaneously
- Token takeover: can attacker use invite meant for someone else?

### B. RBAC
- Privilege escalation: can member become admin without permission?
- Owner transfer hijacking: can attacker intercept transfer?
- Self-demotion: can owner demote themselves to member?
- Role confusion: admin listing after demotion → still has permissions?

### C. Team Visibility
- Information disclosure: 404 vs 403 timing attack
- Package enumeration: can outsider iterate and find team packages?
- Download URL guessability: can non-member guess download URL?

### D. Deprecation
- Reason field injection: XSS if reason rendered in web UI without escaping
- Mass deprecation: can regular member deprecate team packages?
- DoS: deprecating 1000 versions at once?

### E. Dependency Graph
- Cycle attack: self-referencing packages for DoS
- Recursion depth: does deep dependency tree cause stack overflow?
- Fake package injection: does the resolver trust manifest_dependencies from DB?

### F. API Tokens
- Token generation: cryptographically secure?
- Exposure: are tokens logged or returned in error messages?
- Revocation: immediate or cached?

### G. User Settings
- Password change: CSRF protection?
- Profile update: can user set arbitrary fields?

## Deliverable

File: `analysis_results/SECURITY_REVIEW.md`

Structure:
- Executive Summary (clean / issues found)
- Finding #N: title, severity (Critical/High/Medium/Low), description, affected component, recommendation, status (fixed/accepted)
- Dependencies audited (list of new dependencies checked with `pip-audit` or `safety`)

## Acceptance Criteria

- [ ] All 7 areas reviewed
- [ ] No Critical or High findings unfixed
- [ ] Review document is complete and accurate
- [ ] `docs/SECURITY.md` updated if any auth model or secrets handling changed

## Validation

```bash
cd apps/server && pip-audit  # check new dependencies
cd apps/server && ruff check app/  # no security-related rules violated
```

## Outputs

- `analysis_results/SECURITY_REVIEW.md` — security review report
- Updated `docs/SECURITY.md` if needed
