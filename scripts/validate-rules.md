# LLM AS Judge 验证脚本

## 验证目标

评估精简后的 `.claude/rules/` 规则文件是否完整、清晰、可操作。

## 验证维度

| 维度 | 权重 | 评估标准 |
|---|---|---|
| 完整性 | 30% | 核心规则是否保留，是否有遗漏 |
| 清晰性 | 25% | 规则是否易于理解，表述是否准确 |
| 可操作性 | 25% | 规则是否可直接应用到代码中 |
| 上下文效率 | 20% | 信息密度是否提升，是否更精炼 |

## 测试场景

### 场景 1: 创建新的 Python API 端点

**任务**: 创建一个新的 `/packages/{name}/versions` 端点，返回包的版本列表。

**期望 LLM 回应**:
- 使用 `APIRouter` 组织路由
- 使用 `Depends(get_db_session)` 注入数据库会话
- 返回 `PaginatedResponse[VersionResponse]` 类型
- 使用 `select` 语句查询数据库
- 处理软删除过滤
- 添加 OpenAPI 文档注释

### 场景 2: 编写 React 组件

**任务**: 创建一个 `PackageCard` 组件，显示包的基本信息。

**期望 LLM 回应**:
- 使用函数组件和 TypeScript
- 定义 `PackageCardProps` 接口
- 使用 `useCallback` 缓存事件处理
- 使用 `useMemo` 缓存计算结果
- 使用 Tailwind CSS 工具类
- 使用 `cn()` 函数处理条件类名

### 场景 3: 提交代码到 Git

**任务**: 提交一个新功能的代码。

**期望 LLM 回应**:
- 使用 Conventional Commits 格式
- 使用 `feat(server):` 或 `feat(web):` 前缀
- 提交消息使用中文
- 每个提交只做一件事

### 场景 4: 编写测试用例

**任务**: 为 `UserService.create_user` 方法编写测试。

**期望 LLM 回应**:
- 使用 AAA 模式（Arrange-Act-Assert）
- 测试正常路径、边界条件、错误处理
- 使用 `pytest.fixture` 共享测试数据
- 使用 `AsyncMock` 模拟异步操作

## 验证流程

1. **准备阶段**
   - 读取精简后的规则文件
   - 准备测试场景

2. **执行阶段**
   - 针对每个测试场景，让 LLM 给出编码建议
   - 记录 LLM 的回应内容

3. **评估阶段**
   - 对比回应与期望
   - 评估完整性、清晰性、可操作性
   - 计算综合得分

4. **优化阶段**
   - 根据评估结果补充遗漏的规则
   - 优化表述不清的规则
   - 重新验证直到满足标准

## 通过标准

- 所有测试场景的建议一致性 ≥ 95%
- 无重要规则遗漏
- LLM 反馈规则表述清晰、可操作
- 综合得分 ≥ 85 分
